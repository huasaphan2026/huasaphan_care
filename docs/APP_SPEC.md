# Hua Saphan Care — Application Specification

เอกสารฉบับนี้เป็นสเปกหลักของเว็บแอป `Hua Saphan Care` ใช้ควบคุมแนวทางการสร้างระบบให้ตรงกับแผนใน `README.md` โดยเน้นภาพรวมของแอป ขอบเขตฟังก์ชัน บทบาทผู้ใช้ กฎการทำงาน และเกณฑ์ยอมรับของ MVP

> ไฟล์นี้ไม่ใช่ไฟล์ออกแบบฐานข้อมูลเต็ม ไม่ใช่เอกสาร API เต็ม และไม่ใช่คู่มือ UI รายละเอียดเชิงหน้าจอ  
> รายละเอียดเชิงลึกจะแยกไปอยู่ในไฟล์:
>
> - `UI_FLOW.md`
> - `DATA_SCHEMA.md`
> - `API_SPEC.md`
> - `DEVELOPMENT_RULES.md`

## 1. Product Overview

### 1.1 ชื่อระบบ

```txt
Hua Saphan Care
```

### 1.2 URL เป้าหมาย

```txt
https://huasaphan-care.pages.dev
```

### 1.3 พื้นที่ใช้งาน

```txt
บ้านหัวสะพาน หมู่ที่ 1
ตำบลเวียงสระ
อำเภอเวียงสระ
จังหวัดสุราษฎร์ธานี
```

### 1.4 วัตถุประสงค์ของระบบ

`Hua Saphan Care` เป็นเว็บแอปสำหรับรับเรื่อง แจ้งปัญหา ติดตามสถานะ และสรุปผลการแก้ไขปัญหาของชุมชนบ้านหัวสะพาน โดยออกแบบให้ชาวบ้านใช้งานง่ายผ่านมือถือ และให้ผู้นำชุมชนหรือผู้ดูแลระบบสามารถจัดการเรื่องร้องเรียนได้เป็นระบบมากขึ้น

ระบบนี้ต้องช่วยให้ชุมชนมีเครื่องมือกลางในการ:

- รับเรื่องปัญหาจากชาวบ้าน
- เปิดโอกาสให้แจ้งแบบไม่ระบุชื่อ
- แนบภาพประกอบปัญหา
- สร้างรหัสติดตามเรื่อง
- ติดตามสถานะการดำเนินงาน
- มอบหมายผู้รับผิดชอบ
- บันทึกความคืบหน้าเป็น timeline
- แสดงผลงานที่แก้ไขแล้วแบบคัดกรองข้อมูลส่วนตัว
- ส่งออกข้อมูลเพื่อใช้ประชุมหรือประกอบการประเมิน

### 1.5 แนวคิดหลักของระบบ

```txt
เริ่มง่าย ใช้จริงก่อน ขยายทีหลัง
```

ระบบต้องเริ่มจาก MVP ที่ทำงานได้จริง ไม่ซับซ้อนเกินจำเป็น และไม่เพิ่มฟีเจอร์ที่ทำให้การพัฒนาเริ่มต้นยากโดยไม่จำเป็น

## 2. Technical Direction

### 2.1 Stack ที่เลือกใช้

```txt
Frontend: Cloudflare Pages + HTML/CSS/JavaScript
Backend/API: Cloudflare Pages Functions
Database: Cloudflare D1
File Storage: Cloudflare R2
Repository: GitHub repo เดียว
```

### 2.2 หลักการด้านเทคนิค

- ใช้ HTML/CSS/JavaScript ธรรมดาในช่วงเริ่มต้น
- ยังไม่ใช้ React, Vite หรือ framework หนักใน MVP
- ใช้ Pages Functions เป็น API หลังบ้าน
- ใช้ D1 เก็บข้อมูลเชิงตาราง
- ใช้ R2 เก็บรูปภาพและไฟล์แนบ
- ไม่เก็บไฟล์รูปภาพ binary ลงฐานข้อมูล
- ใช้ GitHub repo เดียวควบคุมทั้ง frontend, API, schema และ docs
- ทำระบบ Admin Login เองแบบง่ายก่อน
- ใช้ session cookie สำหรับป้องกันหน้า Admin
- รายงานเริ่มต้นใช้ CSV ก่อน ยังไม่ทำ PDF อัตโนมัติ

### 2.3 สิ่งที่ยังไม่ทำใน MVP

- ยังไม่ทำ real-time notification
- ยังไม่ทำระบบ chat
- ยังไม่ทำ mobile native app
- ยังไม่ใช้ React/Vite
- ยังไม่แยก backend เป็น Workers project แยก
- ยังไม่ทำ PDF generation หนักบน API
- ยังไม่ทำระบบ permission ซับซ้อนหลายชั้น
- ยังไม่เปิดไฟล์ R2 เป็น public ทั้งหมด
- ยังไม่เก็บข้อมูลส่วนบุคคลเกินความจำเป็น
- ยังไม่ทำระบบแผนที่ปัญหาแบบเต็มรูปแบบ

## 3. Target Users

### 3.1 ชาวบ้าน / ผู้แจ้งเรื่อง

ผู้ใช้กลุ่มหลักของระบบ คือ ชาวบ้านในพื้นที่บ้านหัวสะพานที่ต้องการแจ้งปัญหา ข้อเสนอแนะ หรือความเดือดร้อนในชุมชน

ความต้องการสำคัญ:

- ใช้งานผ่านมือถือได้ง่าย
- ไม่ต้องสมัครสมาชิก
- แจ้งเรื่องได้รวดเร็ว
- แนบรูปภาพได้
- แจ้งแบบไม่ระบุชื่อได้
- ได้รหัสติดตามหลังส่งเรื่อง
- ตรวจสอบสถานะย้อนหลังได้

### 3.2 Admin / ผู้นำชุมชน / ผู้ดูแลระบบ

ผู้ดูแลระบบใช้สำหรับรับเรื่อง ตรวจสอบรายละเอียด จัดลำดับความสำคัญ มอบหมายงาน และอัปเดตสถานะ

ความต้องการสำคัญ:

- Login เข้าระบบได้
- เห็นเรื่องใหม่และเรื่องเร่งด่วนก่อน
- ดูรายละเอียดเรื่องและรูปแนบได้
- เปลี่ยนสถานะได้
- เพิ่ม timeline ได้
- มอบหมายผู้รับผิดชอบได้
- เลือกเรื่องที่แก้ไขแล้วไปเผยแพร่เป็นผลงานชุมชนได้
- ส่งออกข้อมูล CSV ได้

### 3.3 ผู้รับผิดชอบงาน / Staff

ผู้รับผิดชอบงานใช้สำหรับดูเรื่องที่ได้รับมอบหมายและอัปเดตความคืบหน้า

ความต้องการสำคัญ:

- เห็นงานที่ได้รับมอบหมาย
- เพิ่มบันทึกความคืบหน้า
- แนบภาพหลังดำเนินการ
- แจ้งสถานะการแก้ไข

### 3.4 ผู้ดูรายงาน / Viewer

ผู้ดูรายงานมีสิทธิ์ดูข้อมูลสรุป แต่ไม่ควรเห็นข้อมูลส่วนบุคคลเต็มรูปแบบ

ความต้องการสำคัญ:

- ดู dashboard ได้
- ดูรายงานสรุปได้
- เห็นข้อมูลเชิงสถิติ
- ไม่สามารถแก้ไขข้อมูลสำคัญ

## 4. User Roles and Permissions

| Role | ชื่อบทบาท | สิทธิ์หลัก |
|---|---|---|
| `super_admin` | ผู้ดูแลหลัก | จัดการทุกอย่าง ผู้ใช้ หมวด รายงาน ตั้งค่า แก้ไขข้อมูลสำคัญ |
| `admin` | ผู้ดูแลระบบ | ดูทุกเรื่อง รับเรื่อง เปลี่ยนสถานะ มอบหมายงาน เพิ่มข่าว/ประกาศ |
| `staff` | ผู้รับผิดชอบงาน | ดูเรื่องที่ได้รับมอบหมาย อัปเดตความคืบหน้า แนบภาพหลังดำเนินการ |
| `viewer` | ผู้ดูรายงาน | ดู Dashboard และรายงาน ไม่เห็นข้อมูลอ่อนไหวเต็มรูปแบบ |

### 4.1 กฎสิทธิ์พื้นฐาน

- `super_admin` เห็นและจัดการข้อมูลได้ทุกส่วน
- `admin` เห็นเรื่องทั้งหมดและจัดการ workflow ได้
- `staff` เห็นเฉพาะงานที่เกี่ยวข้องหรือได้รับมอบหมาย
- `viewer` ดูรายงานได้เท่านั้น ไม่สามารถแก้ไขข้อมูล
- API ฝั่ง Admin ทุก endpoint ต้องตรวจ session และ role
- หน้า `/admin/*` ต้องถูกป้องกันเสมอ

## 6. Public User Features

### 6.1 หน้าแรก

ไฟล์:

```txt
public/index.html
```

หน้าที่:

- แสดงชื่อระบบ `Hua Saphan Care`
- อธิบายระบบแบบสั้น
- มีปุ่มหลัก “แจ้งเรื่องใหม่”
- มีปุ่ม “ติดตามสถานะ”
- มีปุ่มหรือพื้นที่แสดงเบอร์ติดต่อฉุกเฉิน
- มีลิงก์ไปยังผลงานที่แก้ไขแล้ว
- แสดงคำเตือนว่าไม่ใช่ระบบแจ้งเหตุฉุกเฉินแทนหน่วยงานฉุกเฉิน

### 6.2 แบบฟอร์มแจ้งเรื่อง

ไฟล์:

```txt
public/report.html
```

ผู้ใช้ต้องสามารถกรอกข้อมูลต่อไปนี้:

| ข้อมูล | บังคับ | หมายเหตุ |
|---|---:|---|
| หมวดปัญหา | ใช่ | ดึงจาก D1 |
| หัวข้อเรื่อง | ใช่ | สั้น กระชับ |
| รายละเอียดปัญหา | ใช่ | อธิบายสิ่งที่เกิดขึ้น |
| จุดเกิดเหตุ / จุดสังเกต | ใช่ | ไม่ควรบังคับบ้านเลขที่ |
| พิกัดตำแหน่ง | ไม่ | ขออนุญาตผู้ใช้ก่อน |
| รูปภาพ | ไม่ | สูงสุด 3 ภาพ |
| ไม่ประสงค์เปิดเผยชื่อ | ใช่ | ค่าเริ่มต้นควรเปิดเป็น anonymous |
| ชื่อผู้แจ้ง | ไม่ | กรอกเฉพาะเมื่อไม่ anonymous |
| เบอร์โทร | ไม่ | ใช้ติดต่อกลับ ไม่แสดงสาธารณะ |

### 6.3 การแจ้งแบบไม่ระบุชื่อ

กฎ:

- ค่าเริ่มต้นของแบบฟอร์มควรเป็น “ไม่ประสงค์เปิดเผยชื่อ”
- ถ้าเลือก anonymous ไม่ต้องกรอกชื่อและเบอร์โทร
- ระบบยังต้องสร้าง tracking code ให้ติดตามได้
- Admin เห็นรายละเอียดปัญหาและพื้นที่ แต่ไม่เห็นชื่อ/เบอร์ที่ไม่มีการกรอก
- ถ้าเป็นเรื่องฉุกเฉิน ต้องแนะนำให้โทรเบอร์ฉุกเฉินโดยตรง

### 6.4 การแนบภาพ

กฎ:

```txt
สูงสุด 3 ภาพต่อเรื่องสำหรับฝั่งชาวบ้าน
ไฟล์ต้นฉบับไม่เกิน 8 MB ต่อไฟล์
ชนิดไฟล์ที่ยอมรับ: jpg, jpeg, png, webp
ความกว้างหลังบีบอัดไม่เกิน 1280 px
ขนาดเป้าหมายหลังบีบอัดไม่เกิน 500 KB ต่อภาพ
```

ระบบต้อง:

- ตรวจจำนวนภาพก่อนอัปโหลด
- ตรวจชนิดไฟล์
- ตรวจขนาดไฟล์
- บีบอัดภาพฝั่ง browser ก่อนส่ง
- แสดง preview ก่อนส่ง
- แจ้งเตือนถ้าภาพเกินจำนวนหรือขนาด
- บันทึกไฟล์จริงใน R2
- บันทึก metadata ใน D1

### 6.5 หน้าส่งเรื่องสำเร็จ

ไฟล์:

```txt
public/success.html
```

ต้องแสดง:

- ข้อความว่าส่งเรื่องสำเร็จ
- รหัสติดตาม เช่น `HSC-2026-0001`
- ปุ่มคัดลอกรหัส
- ปุ่มไปหน้าติดตามสถานะ
- ข้อความเตือนให้บันทึกรหัสไว้

### 6.6 หน้าติดตามสถานะ

ไฟล์:

```txt
public/track.html
```

ผู้ใช้ต้องสามารถ:

- กรอกรหัสติดตาม
- ดูสถานะล่าสุด
- ดู timeline ที่เปิดเผยต่อสาธารณะ
- ดูข้อความอัปเดตจาก Admin
- ไม่เห็นข้อมูลส่วนตัวของผู้แจ้งหรือผู้ถูกร้องเรียน
- เห็นข้อความแนะนำเมื่อกรอกรหัสผิด

### 6.7 หน้าผลงานชุมชน

ไฟล์:

```txt
public/works.html
```

ต้องแสดงเฉพาะเรื่องที่:

```txt
status = resolved หรือ closed
public_visible = 1
```

ข้อมูลที่แสดงได้:

- หมวดปัญหา
- สรุปปัญหาแบบไม่ระบุตัวบุคคล
- พื้นที่แบบคร่าว ๆ
- สถานะ “แก้ไขแล้ว”
- วันที่แจ้งหรือวันที่แก้ไข
- ภาพก่อน-หลังเฉพาะที่ Admin อนุญาต

ข้อมูลที่ห้ามแสดง:

- ชื่อผู้แจ้ง
- เบอร์โทร
- บ้านเลขที่
- พิกัดละเอียด
- ชื่อผู้ถูกร้องเรียน
- รายละเอียดข้อขัดแย้งส่วนบุคคล
- ภาพที่เห็นใบหน้า ทะเบียนรถ หรือข้อมูลระบุตัวบุคคล

## 7. Admin Features

### 7.1 Admin Login

ไฟล์:

```txt
public/admin/login.html
```

ระบบต้อง:

- รับ username และ password
- ตรวจ password จาก hash เท่านั้น
- สร้าง session cookie
- มีปุ่ม logout
- ไม่เก็บรหัสผ่านจริงใน GitHub
- ป้องกัน brute force อย่างง่ายหลัง MVP
- ตรวจ session ก่อนเข้าหน้า `/admin/*`

### 7.2 Admin Dashboard

ไฟล์:

```txt
public/admin/dashboard.html
```

ต้องแสดงข้อมูลหลัก:

- จำนวนเรื่องใหม่
- จำนวนเรื่องเร่งด่วน
- จำนวนเรื่องกำลังดำเนินการ
- จำนวนเรื่องแก้ไขแล้ว
- จำนวนเรื่องตามหมวด
- รายการเรื่องล่าสุด
- รายการเรื่องเร่งด่วนที่ยังไม่ปิด

กฎ:

- Dashboard ต้อง query แบบสรุป ไม่ดึงทั้งตารางโดยไม่จำเป็น
- เรื่องใหม่และเรื่องเร่งด่วนต้องเห็นก่อน
- ต้องรองรับการแสดงผลบนมือถือและเดสก์ท็อป

### 7.3 รายการเรื่องทั้งหมด

ไฟล์:

```txt
public/admin/reports.html
```

ต้องมี:

- รายการเรื่องแบบแบ่งหน้า
- ค้นหาด้วย tracking code
- ค้นหาด้วยหัวข้อ
- ค้นหาด้วยพื้นที่
- กรองตามสถานะ
- กรองตามหมวด
- กรองตามความเร่งด่วน
- กดเข้าไปดูรายละเอียดเรื่องได้

### 7.4 รายละเอียดเรื่อง

ไฟล์:

```txt
public/admin/report-detail.html
```

ต้องแสดง:

- tracking code
- หมวดปัญหา
- หัวข้อเรื่อง
- รายละเอียด
- จุดเกิดเหตุ / จุดสังเกต
- พิกัด ถ้ามี
- รูปภาพแนบ
- สถานะปัจจุบัน
- ผู้รับผิดชอบ
- timeline
- ข้อมูลผู้แจ้ง เฉพาะกรณีผู้แจ้งยินยอมให้ข้อมูล
- ปุ่มเปลี่ยนสถานะ
- ปุ่มเพิ่ม timeline
- ปุ่มมอบหมายงาน
- ปุ่มตั้งค่าการเผยแพร่เป็นผลงานชุมชน
- ปุ่มแนบภาพหลังดำเนินการ

### 7.5 การเปลี่ยนสถานะ

Admin ต้องสามารถเปลี่ยนสถานะได้ตามรายการ:

| Status | ชื่อไทย | ใช้เมื่อ |
|---|---|---|
| `new` | เรื่องใหม่ | เรื่องที่เพิ่งส่งเข้ามา |
| `accepted` | รับเรื่องแล้ว | Admin ตรวจและรับเรื่อง |
| `checking` | ตรวจสอบข้อมูล | ต้องตรวจสอบพื้นที่หรือข้อมูลเพิ่ม |
| `in_progress` | กำลังดำเนินการ | เริ่มแก้ไขแล้ว |
| `forwarded` | ส่งต่อหน่วยงาน | ส่งต่อ อบต. ไฟฟ้า ตำรวจ หรือหน่วยงานอื่น |
| `waiting_info` | รอข้อมูลเพิ่มเติม | ต้องการข้อมูลเพิ่ม |
| `resolved` | แก้ไขแล้ว | ดำเนินการเสร็จแล้ว |
| `closed` | ปิดเรื่อง | ปิดงานและเก็บประวัติ |
| `rejected` | ไม่รับดำเนินการ | ไม่อยู่ในขอบเขตหรือข้อมูลไม่พอ |

กฎ:

- ทุกครั้งที่เปลี่ยนสถานะ ต้องบันทึก `report_updates`
- ทุกครั้งที่เปลี่ยนสถานะ ต้องบันทึก `audit_logs`
- สถานะที่แสดงในหน้าสาธารณะต้องผ่านการคัดกรองข้อความ
- ถ้า `rejected` ต้องมีเหตุผลประกอบ

### 7.6 Timeline

Timeline ใช้แสดงประวัติความคืบหน้าของเรื่อง

แต่ละรายการควรมี:

- วันที่และเวลา
- สถานะ
- ข้อความอัปเดต
- ผู้บันทึก
- flag ว่าแสดงสาธารณะหรือไม่

กฎ:

- ข้อความที่เปิดเผยต่อผู้ใช้ต้องไม่หลุดข้อมูลส่วนตัว
- Admin สามารถกำหนดได้ว่า update ใดเป็น public หรือ internal
- Public track page แสดงเฉพาะ update ที่ `is_public = 1`

### 7.7 การมอบหมายงาน

Admin ต้องสามารถ:

- เลือกผู้รับผิดชอบ
- กำหนดวันครบกำหนดได้ถ้ามี
- ใส่หมายเหตุภายใน
- เปลี่ยนผู้รับผิดชอบได้
- บันทึก audit log

### 7.8 การเผยแพร่เป็นผลงานชุมชน

Admin ต้องเลือกเผยแพร่เองเท่านั้น

ค่าเริ่มต้น:

```txt
public_visible = 0
```

ก่อนเผยแพร่ Admin ต้องตรวจ:

- ไม่มีชื่อผู้แจ้ง
- ไม่มีเบอร์โทร
- ไม่มีบ้านเลขที่
- ไม่มีพิกัดละเอียด
- ไม่มีชื่อผู้ถูกร้องเรียน
- ไม่มีภาพติดใบหน้า ทะเบียนรถ หรือข้อมูลส่วนตัว
- สรุปเนื้อหาเป็นภาษากลาง ไม่ชี้ตัวบุคคล

## 8. Categories

หมวดปัญหาเริ่มต้นมี 8 หมวด

| Code | หมวด | ความเร่งด่วนเริ่มต้น |
|---|---|---|
| `electric` | ไฟฟ้า / ไฟทาง | normal |
| `road` | ถนน / ทางเดิน | normal |
| `water` | น้ำ / ท่อระบายน้ำ | normal |
| `waste` | ขยะ / ความสะอาด | normal |
| `safety` | ความปลอดภัย | urgent |
| `nuisance` | เหตุรำคาญ | normal |
| `health` | สุขภาพ / ผู้เปราะบาง | urgent |
| `suggestion` | ข้อเสนอแนะ | normal |

### 8.1 กฎหมวดปัญหา

- หมวดต้องดึงจากฐานข้อมูล ไม่ hardcode ในหน้าเว็บ
- Admin สามารถปิดการใช้งานหมวดได้
- ไม่ควรลบหมวดที่เคยมี report แล้ว
- ถ้าต้องการซ่อนหมวด ให้เปลี่ยน `active = 0`
- เวอร์ชันแรกไม่ควรแยกหมวดย่อยมากเกินไป

1. Setup repo และโครงสร้างไฟล์
2. สร้าง `schema.sql` และ `seed.sql`
3. สร้าง `/api/health`
4. สร้าง `/api/public/categories`
5. สร้างหน้าแรก `index.html`
6. สร้างแบบฟอร์ม `report.html`
7. สร้าง `POST /api/reports`
8. เพิ่มระบบบีบอัดและอัปโหลดภาพ
9. สร้าง `success.html`
10. สร้าง `track.html`
11. สร้าง `/api/reports/track`
12. สร้าง Admin Login
13. สร้าง Admin Dashboard
14. สร้าง Admin Reports List
15. สร้าง Admin Report Detail
16. เพิ่ม status update / timeline / assignment
17. เพิ่ม public works
18. เพิ่ม CSV export
19. เพิ่ม PWA พื้นฐาน
20. ทดสอบและแก้ bug

---

## 20. Change Control

หากต้องการเพิ่มฟีเจอร์ใหม่ ต้องตรวจ 4 ข้อก่อน:

1. ฟีเจอร์นี้จำเป็นต่อ MVP หรือไม่
2. ทำให้ระบบซับซ้อนขึ้นเกินจำเป็นหรือไม่
3. กระทบความปลอดภัยหรือข้อมูลส่วนตัวหรือไม่
4. กระทบค่าใช้จ่าย Cloudflare หรือไม่

ถ้าไม่จำเป็นต่อ MVP ให้ย้ายไปไว้ใน backlog หลัง MVP

---

## 21. Backlog หลัง MVP

รายการที่พิจารณาเพิ่มภายหลัง:

- ระบบแผนที่ปัญหา
- QR Code สำหรับแจ้งเรื่อง
- notification ผ่าน LINE หรือ Email
- PDF report
- chart รายเดือน
- SLA ตามหมวดปัญหา
- backup/export อัตโนมัติ
- Cloudflare Access สำหรับ Admin ระดับสูง
- React/Vite หากระบบซับซ้อนขึ้นจริง
- ระบบหลายหมู่บ้านในอนาคต
