# Hua Saphan Care — UI Flow Specification

เอกสารฉบับนี้ใช้ควบคุมแนวทางการออกแบบลำดับหน้าจอ การนำทาง และประสบการณ์ใช้งานของเว็บแอป `Hua Saphan Care` ให้ตรงตามแผนใน `README.md` และ `APP_SPEC.md`

> เป้าหมายของไฟล์นี้คือทำให้ผู้พัฒนา หรือ AI Coding เช่น Codex เข้าใจว่าแต่ละหน้าต้องเชื่อมกันอย่างไร ผู้ใช้เริ่มจากจุดไหน กดอะไรแล้วไปหน้าไหน ข้อมูลใดต้องแสดง และสถานะใดต้องเกิดขึ้นในแต่ละ flow

## 1. หลักการออกแบบ UI / UX

### 1.1 แนวทางหลัก

```txt
mobile-first
clean
friendly
non-governmental
pastel
card-based
ง่ายสำหรับชาวบ้าน
ชัดเจนสำหรับผู้ดูแลระบบ
```

### 1.2 กลุ่มผู้ใช้หลัก

ระบบมีผู้ใช้ 2 กลุ่มใหญ่:

1. **ชาวบ้าน / ผู้แจ้งเรื่อง**
   - ใช้งานผ่านมือถือเป็นหลัก
   - ไม่จำเป็นต้องสมัครสมาชิก
   - แจ้งเรื่องได้ทั้งแบบระบุชื่อและไม่ระบุชื่อ
   - ต้องได้รับรหัสติดตามหลังส่งเรื่อง

2. **ผู้ดูแลระบบ / กรรมการ / ผู้รับผิดชอบงาน**
   - ต้อง login ก่อนเข้าใช้งาน
   - จัดการเรื่องร้องเรียน เปลี่ยนสถานะ มอบหมายงาน เพิ่ม timeline และส่งออกรายงาน

### 1.3 หลัก UX ที่ต้องยึด

- หน้าแรกต้องมีปุ่ม `แจ้งเรื่องใหม่` ชัดเจนที่สุด
- แบบฟอร์มต้องสั้น ไม่ถามข้อมูลเกินจำเป็น
- ทุกหน้าต้องอ่านง่ายบนมือถือ
- ปุ่มหลักต้องใหญ่ กดง่าย
- ใช้ภาษาไทยเรียบง่าย ไม่ใช้ศัพท์เทคนิค
- การติดตามสถานะต้องใช้ `tracking_code` เท่านั้น ไม่ต้อง login
- ข้อมูลส่วนตัวต้องไม่แสดงในหน้าสาธารณะ
- หน้า Admin ต้องเน้นงานที่ต้องทำก่อน เช่น เรื่องใหม่ เรื่องเร่งด่วน เรื่องรอดำเนินการ
- ตารางในมือถือควรเปลี่ยนเป็น card list
- ทุก action สำคัญต้องมี confirmation หรือ feedback ชัดเจน

## 2. โครงสร้างหน้าจอทั้งหมด

### 2.1 ฝั่งชาวบ้าน

| ไฟล์ | ชื่อหน้าจอ | หน้าที่หลัก |
|---|---|---|
| `index.html` | หน้าแรก | แนะนำระบบ ปุ่มแจ้งเรื่อง ติดตามสถานะ เบอร์ฉุกเฉิน |
| `report.html` | แจ้งเรื่องใหม่ | แบบฟอร์มรับเรื่อง แนบภาพ ระบุตำแหน่ง ไม่ระบุชื่อ |
| `success.html` | ส่งเรื่องสำเร็จ | แสดงรหัสติดตาม แนะนำให้บันทึก/คัดลอก |
| `track.html` | ติดตามสถานะ | กรอกรหัสติดตาม ดูสถานะและ timeline |
| `works.html` | ผลงานที่แก้ไขแล้ว | แสดงเรื่องที่ Admin อนุญาตเผยแพร่ |
| `emergency.html` | ช่องทางฉุกเฉิน | แสดงเบอร์ฉุกเฉินและคำเตือน |
| `offline.html` | ออฟไลน์ | แจ้งว่าไม่มีอินเทอร์เน็ตและให้ลองใหม่ |

### 2.2 ฝั่ง Admin

| ไฟล์ | ชื่อหน้าจอ | หน้าที่หลัก |
|---|---|---|
| `admin/login.html` | เข้าสู่ระบบ | Login ผู้ดูแลระบบ |
| `admin/dashboard.html` | Dashboard | สรุปภาพรวม เรื่องใหม่ เรื่องเร่งด่วน สถิติ |
| `admin/reports.html` | รายการเรื่อง | ค้นหา กรอง แบ่งหน้า รายการเรื่องทั้งหมด |
| `admin/report-detail.html` | รายละเอียดเรื่อง | ดูรายละเอียด เปลี่ยนสถานะ เพิ่ม timeline มอบหมายงาน |
| `admin/categories.html` | หมวดปัญหา | จัดการหมวดปัญหา |
| `admin/users.html` | ผู้ใช้ระบบ | จัดการ admin/staff/viewer |
| `admin/news.html` | ข่าว/ประกาศ | จัดการข่าวหรือประกาศหน้าแรก |
| `admin/settings.html` | ตั้งค่าระบบ | ตั้งค่าเบอร์ฉุกเฉิน จำนวนภาพ และค่าพื้นฐาน |
| `admin/reports-export.html` | ส่งออกรายงาน | ดาวน์โหลด CSV ตามช่วงวันที่/สถานะ/หมวด |

## 3. Navigation Structure

### 3.1 Public Navigation

เมนูฝั่งชาวบ้านควรเรียบง่าย ไม่ซับซ้อน

```txt
หน้าแรก
├─ แจ้งเรื่องใหม่
├─ ติดตามสถานะ
├─ ผลงานที่แก้ไขแล้ว
└─ ช่องทางฉุกเฉิน
```

### 3.2 Public Header

ทุกหน้าสาธารณะควรมี:

- Logo หรือชื่อ `Hua Saphan Care`
- ปุ่มกลับหน้าแรก หรือเมนูสั้น
- ปุ่มฉุกเฉินแบบเห็นง่ายในหน้าที่เหมาะสม

### 3.3 Public Bottom Action

ในมือถือให้เน้นปุ่มหลักด้านล่างหรือส่วนท้ายหน้าจอ เช่น:

```txt
[แจ้งเรื่องใหม่]
[ติดตามสถานะ]
```

### 3.4 Admin Navigation

เมนู Admin ควรมี sidebar บนจอใหญ่ และ bottom/tab หรือ drawer บนมือถือ

```txt
Dashboard
├─ รายการเรื่อง
├─ หมวดปัญหา
├─ ผู้ใช้ระบบ
├─ ข่าว/ประกาศ
├─ ส่งออกรายงาน
├─ ตั้งค่า
└─ ออกจากระบบ
```

## 4. Public User Flow — หน้าแรก

### 4.1 Entry Point

ผู้ใช้เข้าสู่:

```txt
https://huasaphan-care.pages.dev
```

ไฟล์ที่โหลด:

```txt
public/index.html
```

### 4.2 สิ่งที่ต้องเห็นในหน้าแรก

หน้าแรกต้องแสดง:

- ชื่อระบบ `Hua Saphan Care`
- คำอธิบายสั้น เช่น `ระบบรับเรื่องและแก้ปัญหาชุมชนหัวสะพาน`
- ปุ่มหลัก `แจ้งเรื่องใหม่`
- ปุ่มรอง `ติดตามสถานะ`
- เบอร์ติดต่อฉุกเฉิน `0629617942`
- คำเตือนว่าเว็บแอปนี้ไม่ใช่ระบบแจ้งเหตุฉุกเฉินแทนหน่วยงานโดยตรง
- การ์ดหมวดปัญหาหลัก 8 หมวดแบบอ่านง่าย
- ลิงก์ไปหน้า `ผลงานที่แก้ไขแล้ว`

### 4.3 Flow จากหน้าแรก

```txt
index.html
├─ กด แจ้งเรื่องใหม่ → report.html
├─ กด ติดตามสถานะ → track.html
├─ กด โทรฉุกเฉิน → tel:0629617942
├─ กด ผลงานที่แก้ไขแล้ว → works.html
└─ กด ช่องทางฉุกเฉิน → emergency.html
```

### 4.4 Acceptance Criteria

- ผู้ใช้เห็นปุ่มแจ้งเรื่องโดยไม่ต้องเลื่อนหน้ามาก
- เบอร์ฉุกเฉินต้องกดโทรได้บนมือถือ
- หน้าแรกโหลดได้แม้ API บางส่วนล้มเหลว
- หากโหลดหมวดปัญหาไม่ได้ ให้แสดงข้อความ fallback ที่เข้าใจง่าย

## 5. Public User Flow — แจ้งเรื่องใหม่

### 5.1 Entry Point

```txt
report.html
```

เข้าจาก:

```txt
index.html → แจ้งเรื่องใหม่
```

### 5.2 ลำดับการใช้งาน

```txt
เปิด report.html
        ↓
โหลดหมวดปัญหาจาก /api/public/categories
        ↓
ผู้ใช้เลือกหมวดปัญหา
        ↓
กรอกหัวข้อ / รายละเอียด / พื้นที่หรือจุดสังเกต
        ↓
เลือกแนบภาพ 0-3 ภาพ
        ↓
เลือกแจ้งแบบไม่ระบุชื่อ หรือระบุชื่อ/เบอร์
        ↓
กดส่งเรื่อง
        ↓
ระบบ validate ข้อมูล
        ↓
บีบอัดภาพฝั่ง browser
        ↓
ส่งข้อมูลไป /api/reports
        ↓
หากมีภาพ ส่งภาพไป /api/uploads/image
        ↓
ระบบคืน tracking_code
        ↓
ไป success.html?code=HSC-2026-0001
```

### 5.3 Field ในแบบฟอร์ม

| Field | จำเป็น | หมายเหตุ |
|---|---:|---|
| หมวดปัญหา | จำเป็น | ดึงจาก D1 |
| หัวข้อเรื่อง | จำเป็น | สั้น กระชับ |
| รายละเอียดปัญหา | จำเป็น | จำกัดความยาวเบื้องต้น |
| พื้นที่ / จุดสังเกต | จำเป็น | ใช้แทนที่อยู่เต็ม |
| พิกัด | ไม่บังคับ | ขอ permission ก่อนเท่านั้น |
| รูปภาพ | ไม่บังคับ | สูงสุด 3 ภาพ |
| ไม่ประสงค์เปิดเผยชื่อ | ค่าเริ่มต้นเปิด | anonymous = 1 |
| ชื่อผู้แจ้ง | ไม่บังคับ | แสดงเฉพาะเมื่อไม่ anonymous |
| เบอร์โทร | ไม่บังคับ | ใช้ติดต่อกลับ ไม่แสดงสาธารณะ |

### 5.4 Form Validation

ต้องตรวจอย่างน้อย:

- เลือกหมวดแล้วหรือไม่
- หัวข้อไม่ว่าง
- รายละเอียดไม่ว่าง
- พื้นที่/จุดสังเกตไม่ว่าง
- จำนวนภาพไม่เกิน 3
- ขนาดไฟล์ต้นฉบับไม่เกิน 8MB ต่อภาพ
- ชนิดไฟล์ต้องเป็น `jpg`, `jpeg`, `png`, `webp`
- ถ้าไม่ anonymous และกรอกเบอร์ ต้องตรวจรูปแบบเบอร์เบื้องต้น

### 5.5 Image Upload UX

เมื่อผู้ใช้เลือกภาพ:

- แสดง preview รูป
- แสดงจำนวนภาพ เช่น `2/3 ภาพ`
- มีปุ่มลบภาพแต่ละภาพ
- แจ้งเตือนหากภาพใหญ่เกิน
- แจ้งว่าระบบจะบีบอัดภาพก่อนส่ง

### 5.6 Location UX

ควรมี 2 วิธี:

1. กรอกข้อความเอง เช่น `ซอย 3 ใกล้ศาลาหมู่บ้าน`
2. ปุ่ม `ใช้ตำแหน่งปัจจุบัน` หาก browser รองรับ

หากผู้ใช้ไม่อนุญาต location:

- ไม่ต้อง block การส่งเรื่อง
- ให้ใช้ข้อความพื้นที่แทน

### 5.7 Anonymous UX

ค่าเริ่มต้นควรเป็น:

```txt
ไม่ประสงค์เปิดเผยชื่อ: เปิด
```

เมื่อเปิด anonymous:

- ซ่อนหรือไม่บังคับชื่อ/เบอร์
- แสดงข้อความว่า `ระบบจะแสดงเฉพาะรหัสติดตาม ไม่แสดงชื่อผู้แจ้ง`

เมื่อปิด anonymous:

- แสดงช่องชื่อ/เบอร์
- อธิบายว่าใช้เพื่อให้ผู้ดูแลติดต่อกลับเท่านั้น

### 5.8 Loading State

เมื่อกดส่งเรื่อง:

- ปิดปุ่มส่งชั่วคราว
- แสดงข้อความ `กำลังส่งเรื่อง...`
- ห้ามผู้ใช้กดซ้ำหลายครั้ง

### 5.9 Success State

หากส่งสำเร็จ:

```txt
redirect → success.html?code={tracking_code}
```

### 5.10 Error State

หากส่งไม่สำเร็จ:

- แสดงข้อความเข้าใจง่าย
- ไม่ล้างข้อมูลที่ผู้ใช้กรอกทันที
- ให้ปุ่ม `ลองส่งอีกครั้ง`

ตัวอย่างข้อความ:

```txt
ยังส่งเรื่องไม่สำเร็จ กรุณาตรวจสอบอินเทอร์เน็ตแล้วลองใหม่อีกครั้ง
```

### 5.11 Acceptance Criteria

- ผู้ใช้ส่งเรื่องได้โดยไม่ต้อง login
- ระบบสร้าง tracking code ได้
- ภาพไม่เกิน 3 ภาพ
- ภาพถูกบีบอัดก่อนส่ง
- ถ้า anonymous ต้องไม่บังคับชื่อ/เบอร์
- หลังส่งสำเร็จต้องไปหน้า success พร้อมรหัสติดตาม

## 6. Public User Flow — ส่งเรื่องสำเร็จ

### 6.1 Entry Point

```txt
success.html?code=HSC-2026-0001
```

### 6.2 สิ่งที่ต้องแสดง

- ข้อความ `ส่งเรื่องสำเร็จ`
- รหัสติดตามขนาดใหญ่ อ่านง่าย
- ปุ่ม `คัดลอกรหัส`
- ปุ่ม `ติดตามสถานะ`
- ปุ่ม `กลับหน้าแรก`
- คำแนะนำให้บันทึกรหัสไว้

### 6.3 Flow

```txt
success.html
├─ กด คัดลอกรหัส → copy tracking_code
├─ กด ติดตามสถานะ → track.html?code=HSC-2026-0001
└─ กด กลับหน้าแรก → index.html
```

### 6.4 Empty Code State

หากไม่มี code ใน URL:

- แสดงข้อความว่า `ไม่พบรหัสติดตาม`
- มีปุ่มไปหน้า `track.html`

### 6.5 Acceptance Criteria

- รหัสต้องอ่านง่ายและคัดลอกได้
- ผู้ใช้ต้องเข้าใจว่ารหัสนี้ใช้ติดตามสถานะภายหลัง

## 7. Public User Flow — ติดตามสถานะ

### 7.1 Entry Point

```txt
track.html
track.html?code=HSC-2026-0001
```

### 7.2 ลำดับการใช้งาน

```txt
เปิด track.html
        ↓
ถ้ามี code ใน URL ให้ค้นหาอัตโนมัติ
        ↓
ถ้าไม่มี code ให้แสดงช่องกรอก
        ↓
ผู้ใช้กรอกรหัสติดตาม
        ↓
กด ตรวจสอบสถานะ
        ↓
เรียก /api/reports/track?code={code}
        ↓
แสดงสถานะล่าสุด + timeline แบบ public
```

### 7.3 ข้อมูลที่แสดงได้

- รหัสติดตาม
- หมวดปัญหา
- หัวข้อแบบสรุป
- พื้นที่โดยประมาณ
- สถานะล่าสุด
- วันที่แจ้ง
- Timeline ที่ `is_public = 1`
- ข้อความสรุปผลเมื่อแก้ไขแล้ว

### 7.4 ข้อมูลที่ห้ามแสดง

- ชื่อผู้แจ้ง
- เบอร์โทรผู้แจ้ง
- ชื่อผู้ถูกร้องเรียน
- พิกัดละเอียด
- ข้อมูลสุขภาพหรือข้อมูลส่วนตัวละเอียด
- note ภายในของ Admin
- รายละเอียดที่ตั้งที่ระบุตัวบุคคลได้

### 7.5 Status Display

ควรแสดงสถานะด้วย chip สีและคำไทย เช่น:

| Status | คำที่แสดง |
|---|---|
| `new` | เรื่องใหม่ |
| `accepted` | รับเรื่องแล้ว |
| `checking` | ตรวจสอบข้อมูล |
| `in_progress` | กำลังดำเนินการ |
| `forwarded` | ส่งต่อหน่วยงาน |
| `waiting_info` | รอข้อมูลเพิ่มเติม |
| `resolved` | แก้ไขแล้ว |
| `closed` | ปิดเรื่อง |
| `rejected` | ไม่รับดำเนินการ |

### 7.6 Not Found State

หากไม่พบรหัส:

```txt
ไม่พบรหัสติดตามนี้ กรุณาตรวจสอบรหัสอีกครั้ง
```

## 19. Status Flow ของเรื่องร้องเรียน

### 19.1 Flow หลัก

```txt
new
 ↓
accepted
 ↓
checking
 ↓
in_progress
 ↓
resolved
 ↓
closed
```

### 19.2 Flow ทางเลือก

```txt
new → rejected
accepted → waiting_info
checking → forwarded
forwarded → in_progress
waiting_info → checking
resolved → closed
```

### 19.3 กฎสำคัญ

- เรื่องใหม่ต้องเริ่มที่ `new`
- เมื่อ Admin ตรวจแล้วควรเปลี่ยนเป็น `accepted`
- ถ้าต้องลงพื้นที่ตรวจ ให้ใช้ `checking`
- ถ้าเริ่มแก้แล้วใช้ `in_progress`
- ถ้าแก้เสร็จใช้ `resolved`
- เมื่อจบกระบวนการใช้ `closed`
- ถ้าไม่อยู่ในขอบเขตใช้ `rejected` พร้อมเหตุผล

## 24. UI Flow สำหรับ MVP เท่านั้น

ใน MVP ต้องทำให้ครบก่อน:

1. หน้าแรก
2. แจ้งเรื่องใหม่
3. ส่งเรื่องสำเร็จพร้อมรหัสติดตาม
4. ติดตามสถานะ
5. Admin login
6. Admin dashboard
7. รายการเรื่อง
8. รายละเอียดเรื่อง
9. เปลี่ยนสถานะ
10. เพิ่ม timeline
11. ส่งออก CSV เบื้องต้น

สิ่งที่ยังไม่ควรทำใน UI ระยะแรก:

- real-time notification
- chat สด
- แผนที่ละเอียด
- PDF report ในระบบ
- permission ซับซ้อนหลายชั้น
- dashboard chart ซับซ้อน
- public feed แสดงทุกเรื่องอัตโนมัติ

## 25. Definition of Done ของ UI_FLOW.md

การสร้าง UI ถือว่าตรงตามเอกสารนี้เมื่อ:

- ผู้ใช้แจ้งเรื่องจากมือถือได้จริง
- ผู้ใช้ได้รับ tracking code หลังส่งเรื่อง
- ผู้ใช้ติดตามสถานะด้วย tracking code ได้
- Admin login แล้วเห็น dashboard ได้
- Admin เห็นรายการเรื่องใหม่ได้
- Admin เปิดรายละเอียดเรื่องได้
- Admin เปลี่ยนสถานะและเพิ่ม timeline ได้
- ข้อมูลส่วนตัวไม่แสดงในหน้าสาธารณะ
- หน้า works แสดงเฉพาะเรื่องที่อนุญาตเผยแพร่
- ทุกหน้ามี loading/error/empty state ที่เหมาะสม
- UI ใช้โทน modern, clean, friendly, pastel, mobile-first

## 27. สรุป

`UI_FLOW.md` เป็นเอกสารกำกับเส้นทางการใช้งานของระบบ ไม่ใช่ไฟล์ออกแบบฐานข้อมูลหรือ API รายละเอียด การพัฒนาเว็บแอปต้องใช้ไฟล์นี้ร่วมกับ:

- `APP_SPEC.md` สำหรับขอบเขตระบบ
- `DATA_SCHEMA.md` สำหรับฐานข้อมูล
- `API_SPEC.md` สำหรับ endpoint และ payload
- `DEVELOPMENT_RULES.md` สำหรับกฎการพัฒนาและคุณภาพโค้ด

หลักสำคัญคือ:

```txt
ชาวบ้านต้องแจ้งเรื่องได้ง่าย
Admin ต้องจัดการเรื่องได้เป็นระบบ
ข้อมูลส่วนตัวต้องไม่หลุดสู่สาธารณะ
เริ่มจาก MVP ที่ใช้งานจริงก่อน แล้วค่อยขยาย
```
